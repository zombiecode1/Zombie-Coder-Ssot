🧟 ZombieCoder Agent System

বাংলায় সম্পূর্ণ ডকুমেন্টেশন — কিভাবে কাজ করে, কেন লাগে, এবং কোথায় সীমাবদ্ধতা আছে।

১. এজেন্ট কেন লাগে?

সমস্যা: একজন ডেভেলপার IDE/এডিটরে বসে কাজ করেন। সমস্যা হলো — প্রতিবার Stack Overflow, ডকুমেন্টেশন, বা AI চ্যাটে যেতে হয়। এটি সময় নষ্ট করে এবং ফোকাস ভাঙে।

সমাধান: ZombieCoder Agent — একটি local-first AI সহযোগী যে আপনার প্রোজেক্টের কন্টেক্সট বুঝে, SSOT (Single Source of Truth) থেকে তথ্য নেয়, এবং সরাসরি আপনার এডিটরেই উত্তর দেয়।

কেন MCP/Agent, শুধু Claude/GPT চ্যাট যথেষ্ট না কেন?

বিষয়সাধারণ AI চ্যাটZombieCoder Agent
প্রোজেক্ট কন্টেক্সটনেই — শুধু prompt থেকে বুঝে✅ SSOT.md + RAG থেকে প্রকৃত প্রোজেক্ট তথ্য
টুল এক্সিকিউশনশুধু কথা বলে✅ ফাইল পড়ে, কোড সার্চ করে, কমান্ড চালায়
প্রাইভেসিডেটা ক্লাউডে যায়✅ সবকিছু লোকাল
মেমরিপুরোনো conversation মনে থাকে না✅ Session + Conversation ID দিয়ে মনে রাখে
কস্টপ্রতিটি request বিল✅ ফ্রি মডেল আছে (OpenCode)

৩. এজেন্ট চালু হওয়ার প্রক্রিয়া (Startup Flow)

  1. Server Start: node dist/index.js চালু হয়
  2. Workspace Detection: mcp/mcp.json খুঁজে বের করে কোন ডিরেক্টরি workspace root
  3. State DB: .zombiecoder/state.db তৈরি/লোড হয় (SQLite)
  4. SSOT Generation: SSOT.md না থাকলে auto-generate হয়
  5. Workspace Watcher: Chokidar দিয়ে ফাইল পরিবর্তন monitor করে
  6. Provider Bootstrap: Environment variables থেকে providers খুঁজে বের করে
  7. Runtime Manifest: .zombiecoder/runtime.json লেখা হয় (প্রতি ৫ মিনিটে update)
  8. Agent Ready: POST /v1/agent/chat রেডি

৫. SSOT (Single Source of Truth) — কী এবং কেন

SSOT.md হলো আপনার প্রোজেক্টের একটি auto-generated documentation। এটি:

  • প্রোজেক্টের ফোল্ডার স্ট্রাকচার দেখায়
  • কোন কোন ফাইল আছে সেটা list করে
  • এজেন্টকে প্রোজেক্ট সম্পর্কে জানায়
  • প্রতি ৫ মিনিটে auto-update হয় (file change হলে)

SSOT কিভাবে তৈরি হয়?

1. Server start → setWorkingDirectory() → autoInit: true
2. scanProject() → সব ফাইল + ফোল্ডার scan
3. generateSSOTTemplate() → markdown format এ convert
4. saveSSOT() → .zombiecoder/SSOT.md তে save
5. WorkspaceWatcher → ফাইল পরিবর্তন হলে auto-regenerate

SSOT Update Timing

EventUpdate?Time
Server start✅ Autoতৎক্ষণাৎ
ফাইল add/delete/modify✅ Watcher trigger~1.5s debounce
Manual rescan✅ API callতৎক্ষণাৎ
5 minute interval✅ Runtime.jsonপ্রতি ৫ মিনিটে

৬. Conversation Memory — কিভাবে মনে রাখে

তিনটি ID সিস্টেম

IDকাজউদাহরণ
session_idএডিটর session — একই editor restart পর্যন্ত"opencode-main"
conversation_idএকটি নির্দিষ্ট conversation — persist থাকে"ae76bb6b-d5db-..."
workspace_idপ্রোজেক্ট — একই workspace সবার জন্য"auto:3af7cdb0..."

Conversation Save Flow

1. User message আসে
2. ensureConversation() → conversation table তে entry (না থাকলে)
3. addConversationMessage() → user message save
4. LLM call → response
5. addConversationMessage() → assistant message save
6. Usage track → usage_stats table

Memory Limitation

সীমাবদ্ধতা Agent service শুধু বর্তমান request এর messages array ব্যবহার করে। পুরোনো conversation history automatically inject হয় না। এটি client-side responsibility — editor extension কে আগের messages pass করতে হবে।

কেন MCP tools লাগে?

Agent server (proxi_new) একটি API server — এটি শুধু LLM call করে এবং response দেয়। ফাইল অপারেশন বা কমান্ড execute করার জন্য MCP server লাগে যেটি actual tools provide করে।

// MCP Server (tools provider)
{
  "command": "node",
  "args": ["mcp-server.js"],
  "tools": ["read_file", "write_file", "run_command", "search_code"]
}

// Agent Server (brain)
// → MCP tools discover করে
// → LLM decision নেয় কোন tool call করবে
// → Tool execute হয়
// → Result আবার LLM-কে দেওয়া হয়
// → চূড়ান্ত উত্তর তৈরি

৮. Provider System — কস্ট অপটিমাইজেশন

Free-First Strategy

ProviderGateway সবসময় ফ্রি মডেল আগে try করে:

Scoring Formula:
  baseScore = priority (0-100)
  + freeModelBonus: +50 (ফ্রি মডেলের জন্য)
  + contextFit: +10 (context window ফিট হলে)
  - toolPenalty: -20 (tools লাগলে এবং provider support না করলে)

Provider Priority
