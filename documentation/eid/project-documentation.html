বিশ্লেষণের তারিখ: ২৮ মে, ২০২৬ | বিশ্লেষক: AI ডকুমেন্টেশন এজেন্ট

১। প্রকল্প কী?

এটি একটি ওপেন সোর্স টাইপস্ক্রিপ্ট প্রকল্প (নাম: groq-unified-bridge) যা মূলত দুটি বড় কাজ করে:

  • API ব্রিজ বা প্রক্সি: এটি Groq-এর এলএলএম (Large Language Model) API-কে OpenAI-এর API ফরম্যাটে রূপান্তরিত করে। যে কোনো ক্লায়েন্ট যারা OpenAI SDK ব্যবহার করে, তারা সরাসরি Groq-এর মডেল ব্যবহার করতে পারে — কোনো কোড পরিবর্তন ছাড়াই।
  • এজেন্ট ও RAG সিস্টেম (ZombieCoder): একটি স্বয়ংসম্পূর্ণ AI কোডিং এজেন্ট যার নিজস্ব পরিচয় (identity), প্রকল্প স্ক্যান করার ক্ষমতা, ভেক্টর ইনডেক্সিং, কথোপকথন সংরক্ষণ, এবং ফাইল পরিবর্তন ট্র্যাকিংয়ের ব্যবস্থা আছে।
সহজ ভাষায়: আপনি যদি Python-এ OpenAI-এর ChatGPT API ব্যবহার করে থাকেন, কিন্তু Groq-এর বিনামূল্যের API ব্যবহার করতে চান, তাহলে এই ব্রিজটি আপনার লোকাল মেশিনে বসে যায় এবং আপনার OpenAI-ফরম্যাটের রিকোয়েস্টকে Groq-এর কাছে পাঠিয়ে উত্তর ফিরিয়ে আনে।

২। প্রকল্পের গঠন ও ফাইলসমূহ

২.১ প্রধান ফাইল ও ফোল্ডার

  • src/index.ts — সার্ভারের এন্ট্রি পয়েন্ট (Express.js)
  • src/controllers/ — রিকোয়েস্ট হ্যান্ডলার (OpenAI, Agent, MCP কন্ট্রোলার)
  • src/services/ — মূল লজিক (Groq সার্ভিস, RAG, ভেক্টর ইনডেক্স, স্টেট DB)
  • src/middleware/ — অথেনটিকেশন, লগিং, আইডেন্টিটি হেডার
  • src/routes/ — API রুট ডিফাইনিশন
  • src/types/ — টাইপস্ক্রিপ্ট টাইপ ডিফিনিশন
  • identity.json — এজেন্টের স্থির পরিচয় (immutable identity manifest)
  • .env — পরিবেশ ভেরিয়েবল (GROQ_API_KEY, PORT)

২.২ API এন্ডপয়েন্টসমূহ (সার্ভার চালু হলে)

  • POST /v1/chat/completions — চ্যাট সম্পূর্ণ করা (OpenAI-সামঞ্জস্যপূর্ণ)
  • POST /v1/completions — টেক্সট সম্পূর্ণ করা (লিগ্যাসি)
  • POST /v1/audio/transcriptions — অডিও থেকে টেক্সট
  • POST /v1/embeddings — টেক্সট এম্বেডিং
  • GET /v1/models — উপলব্ধ মডেলের তালিকা
  • GET /dashboard — রিয়েল-টাইম মনিটরিং ড্যাশবোর্ড
  • POST /v1/agent/chat — ZombieCoder এজেন্টের সাথে চ্যাট
  • POST /mcp — JSON-RPC MCP এন্ডপয়েন্ট

৩। এটি কীভাবে কাজ করে? (সহজ টেকনিক্যাল ব্যাখ্যা)

৩.১ API ব্রিজ (প্রধান কাজ)

১. আপনি আপনার অ্যাপ থেকে OpenAI-ফরম্যাটে একটি HTTP অনুরোধ পাঠান (যেমন: curl http://localhost:9999/v1/chat/completions)।
২. সার্ভার অনুরোধটি গ্রহণ করে, Groq SDK-এর ফরম্যাটে রূপান্তর করে।
৩. অনুরোধটি Groq-এর ক্লাউড API-তে পাঠানো হয়।
৪. Groq থেকে উত্তর আসলে, সেটিকে আবার OpenAI-ফরম্যাটে রূপান্তর করে ক্লায়েন্টকে ফেরত দেওয়া হয়।
৫. স্ট্রিমিং মোড থাকলে (stream: true), ডেটা টুকরো টুকরো করে SSE (Server-Sent Events) আকারে পাঠানো হয়।

৩.২ স্মার্ট মডেল রাউটিং

যদি ক্লায়েন্ট model: "auto" পাঠায় বা মডেল না দেয়, তাহলে ব্রিজ নিজেই সিদ্ধান্ত নেয় কোন Groq মডেল ব্যবহার করবে:

  • ছোট ইনপুট (১০০ অক্ষরের কম): দ্রুত মডেল (যেমন: llama-3.1-8b-instant)
  • মাঝারি ইনপুট: ব্যালেন্সড মডেল (যেমন: llama-3.3-70b-versatile)
  • বড় ইনপুট: উচ্চ-ক্ষমতার মডেল

পাশাপাশি এটি প্রতি-মডেল রেট লিমিট (RPM = Request Per Minute, TPM = Token Per Minute) ট্র্যাক করে। যদি কোনো মডেলের TPM লিমিট অতিক্রম করার সম্ভাবনা থাকে, তাহলে স্বয়ংক্রিয়ভাবে বেশি TPM সমর্থন করে এমন অন্য মডেলে রুট করে দেয়।

৩.৩ ZombieCoder এজেন্ট সিস্টেম

এটি একটি AI কোডিং এজেন্ট যার নিজস্ব পরিচয় (identity.json থেকে) এবং প্রকল্প বোঝার ক্ষমতা আছে:

  • SSOT (Single Source of Truth): এজেন্ট যখন কোনো প্রকল্পে অ্যাক্সেস পায়, তখন সে প্রকল্পটি স্ক্যান করে .zombiecoder/SSOT.md নামে একটি ডকুমেন্টেশন ফাইল তৈরি করে।
  • RAG (Retrieval-Augmented Generation): ব্যবহারকারী যখন প্রকল্প সম্পর্কে প্রশ্ন করে, এজেন্ট SSOT.md থেকে প্রাসঙ্গিক তথ্য খুঁজে এনে উত্তরের সাথে যুক্ত করে।
  • ভেক্টর ইনডেক্সিং: উন্নত অনুসন্ধানের জন্য ফাইলগুলোকে ছোট ছোট টুকরো (chunk) করে fastembed দিয়ে এম্বেডিং বানিয়ে SQLite-তে সংরক্ষণ করে।
  • পারমিশন সিস্টেম: এজেন্ট প্রথমে অনুমতি চায়, তারপর প্রকল্প স্ক্যান করতে পারে।
  • ফাইল ওয়াচার: chokidar ব্যবহার করে প্রকল্পের ফাইল পরিবর্তন ট্র্যাক করে এবং স্বয়ংক্রিয়ভাবে SSOT ও ইনডেক্স আপডেট করে।

৩.৪ MCP (Model Context Protocol)

JSON-RPC ভিত্তিক একটি এন্ডপয়েন্ট যা এডিটর বা টুল ক্লায়েন্টদের জন্য ডিজাইন করা হয়েছে। এটি কনভার্সেশন তৈরি, হিস্টোরি দেখা, ওয়ার্কস্পেস সার্চ, SSOT পড়া — এই ধরনের কাজ করে। প্রোটোকল ভার্সন 2025-03-26 অনুসরণ করে।

৪। বাস্তব জীবনে ব্যবহার

🛠️ ব্যবহারের ক্ষেত্র
  • ডেভেলপার টুল: যারা ইতিমধ্যে OpenAI SDK ব্যবহার করেন (LangChain, LlamaIndex, custom scripts), তারা Groq-এর বিনামূল্যের API-তে স্যুইচ করতে পারেন শুধু base_url পরিবর্তন করে।
  • স্থানীয় AI কোডিং অ্যাসিস্ট্যান্ট: ZombieCoder এজেন্ট আপনার লোকাল প্রকল্প বুঝতে পারে, প্রশ্নের উত্তর দিতে পারে, এমনকি পারমিশন নিয়ে কোড পরিবর্তন করতে পারে।
  • ভারেন্ডর লক-ইন এড়ানো: Groq-এর পরিবর্তে ভবিষ্যতে অন্য কোনো API ব্যবহার করতে চাইলে, শুধু ব্যাকএন্ড পরিবর্তন করতে হবে — ক্লায়েন্ট সাইড অপরিবর্তিত থাকবে।
  • লোকাল ড্যাশবোর্ড ও মনিটরিং: API ব্যবহারের পরিসংখ্যান, রেট লিমিট স্ট্যাটাস, লাইভ লগ দেখা যায় http://localhost:9999/dashboard-এ।

৫। সত্যতা ও বিশ্লেষণ (Authenticity Assessment)

নিচের বিশ্লেষণটি প্রকল্পের প্রতিটি স্তর যাচাই করে তৈরি করা হয়েছে — কোনো মনগড়া তথ্য নয়, বরং সোর্স কোড পড়ে সিদ্ধান্ত:

উপাদান অবস্থা মন্তব্য
Express.js সার্ভার বাস্তব সম্পূর্ণ কার্যকরী কোড, dist/ ফোল্ডারে কম্পাইলড আউটপুট বিদ্যমান
Groq API ইন্টিগ্রেশন বাস্তব Groq SDK (groq-sdk v0.5.0) সঠিকভাবে ইমপ্লিমেন্ট করা। .env-তে বৈধ API কী আছে
OpenAI ফরম্যাট কনভার্শন বাস্তব সম্পূর্ণ OpenAI-সামঞ্জস্যপূর্ণ রেসপন্স ফরম্যাট (চாயস, ডেল্টা, স্ট্রিমিং, টুল কলিং)
SQLite স্টেট DB বাস্তব better-sqlite3 সঠিকভাবে ব্যবহৃত, WAL মোড, ৭টি টেবিল সহ সম্পূর্ণ স্কিমা
fastembed ভেক্টর ইনডেক্স আংশিক কোড সঠিক, কিন্তু fastembed লাইব্রেরি সিস্টেমে মডেল ডাউনলোড করতে পারে বা নাও পারে। fallback embedder দেওয়া আছে
ZombieCoder আইডেন্টিটি বাস্তব Sahon Srabon (ঢাকা, বাংলাদেশ) নামে একজন বাস্তব ডেভেলপারের পরিচয়, ফোন ও ইমেইল সহ
মডেল তালিকা (যেমন gpt-oss-20b) আংশিক/অনিশ্চিত কিছু মডেল (যেমন 'openai/gpt-oss-20b', 'groq/compound') Groq-এর অফিসিয়াল ডকুমেন্টেশনে নেই। এগুলো পুরনো বা অভ্যন্তরীণ মডেল আইডি হতে পারে
MCP প্রোটোকল আংশিক কাস্টম JSON-RPC ইমপ্লিমেন্টেশন। MCP স্ট্যান্ডার্ডের সাথে পুরোপুরি মিল নাও থাকতে পারে, মুলত নিজস্ব ফরম্যাট
টেস্ট ফাইল বাস্তব api-test.ts, generate-model-report.ts সহ একাধিক টেস্ট ফাইল বিদ্যমান
ফাইল ওয়াচার (chokidar) বাস্তব chokidar v4.0.3 ব্যবহার করে সম্পূর্ণ কার্যকরী ফাইল মনিটরিং

৬। কোন অংশগুলো কতটুকু নির্ভরযোগ্য

✅ সম্পূর্ণ নির্ভরযোগ্য (বাস্তব ও কার্যকরী):
Express সার্ভার, API এন্ডপয়েন্ট, Groq SDK ইন্টিগ্রেশন, SQLite DB, ফাইল লগিং, অথেনটিকেশন, ড্যাশবোর্ড, স্মার্ট রাউটিং — এগুলো সবই বাস্তব ও ব্যবহারযোগ্য।
⚠️ আংশিক নির্ভরযোগ্য (কাজ করে তবে সীমাবদ্ধতা আছে):
  • মডেল রেট লিমিট: হার্ডকোডেড RPM/TPM মান দেয়া আছে, কিন্তু Groq আপনার অ্যাকাউন্টের প্রকৃত লিমিট এর থেকে ভিন্ন হতে পারে। প্রোজেক্ট env ভেরিয়েবলের মাধ্যমে ওভাররাইডের ব্যবস্থা রেখেছে।
  • ভেক্টর এম্বেডিং: fastembed কাজ করলে দারুণ, কিন্তু fallback embedder (SHA-256 ভিত্তিক) আসলে কোনো অর্থপূর্ণ সার্চ ফলাফল দেয় না — এটি একটি placeholder মাত্র।
  • মডেল ক্যাটাগরি: কিছু মডেলের ক্যাটাগরি (যেমন context_window, category) আনুমানিক বা ধারণার ভিত্তিতে সেট করা। Groq API থেকে প্রাপ্ত অ্যাকচুয়াল ডেটা থাকলে সেটি ব্যবহার করা হয়, না থাকলে হার্ডকোডেড মান ব্যবহার করা হয়।
❌ সতর্কতা/অপ্রমাণিত অংশ:
  • "openai/gpt-oss-20b" ও "groq/compound" মডেল: এগুলোর অস্তিত্ব Groq-এর পাবলিক API-তে নিশ্চিত নয়। যদি এগুলো না থাকে, তাহলে অটো-রাউটিং ডিফল্ট মডেলে চলে যাবে — কাজ বন্ধ হবে না, তবে রাউটিং অকার্যকর হবে।
  • MCP প্রোটোকল: এটি একটি কাস্টম ইমপ্লিমেন্টেশন। বাস্তব MCP-সামঞ্জস্যপূর্ণ টুল (যেমন: VS Code extensions) এর সাথে কাজ নাও করতে পারে।
  • লিগ্যাসি এজেন্ট রেসপন্স ফরম্যাট: JSON র‍্যাপার ফরম্যাট (flags, type, execute ইত্যাদি) শুধুমাত্র এই প্রকল্পের নিজস্ব ক্লায়েন্টের জন্য — বাইরের কেউ এটি ব্যবহার করলে বিভ্রান্ত হতে পারে।

৭। পূর্ববর্তী ব্যবহার ও বর্তমান অবস্থা

প্রোজেক্টটির কোড দেখে বোঝা যায় এটি একটি পরিণত কিন্তু উন্নয়নশীল প্রোজেক্ট:

  • কম্পাইলড আউটপুট (dist/) বিদ্যমান: প্রমাণ করে এটি আগেও চালানো হয়েছে এবং বিল্ড সফল হয়েছে।
  • লগ ফাইল (logs/) বিদ্যমান: অন্তত একবার সার্ভার চালু করা হয়েছে এবং রিকোয়েস্ট পাঠানো হয়েছে।
  • টেস্ট ফাইল ও রিপোর্ট: mcp-status-report.json এবং model-response-report.json — এগুলো আগের রানের আউটপুট।
  • .env-এ API কী বিদ্যমান: GROQ_API_KEY পূর্ণ এবং এটি "gsk_" দিয়ে শুরু, যা Groq-এর API কী ফরম্যাট।
  • কিছু অংশ অসম্পূর্ণ: যেমন কিছু কনসোল লগে test messages রয়েছে, যা নির্দেশ করে ডেভেলপমেন্ট পর্যায়ে কাজ চলছে।

সুতরাং, এই প্রোজেক্টটি ব্যক্তিগত বা টিম ডেভেলপমেন্ট পরিবেশে ব্যবহার করা হয়েছে, কিন্তু এটি প্রোডাকশন-গ্রেড নয় — বরং একটি শক্তিশালী টুল যা ডেভেলপারদের জন্য Groq API-কে আরও সহজলভ্য করে তোলে।
