১. মূল ধারণা — সহজ ভাষায়

শুধু কথায় বলি

আমাদের সিস্টেম একটি টেলিফোন এক্সচেঞ্জের মতো। আপনি যখন ফোন করেন, তখন অপারেটর আপনাকে সঠিক নম্বরে কনেক্ট করেন। আমাদের সিস্টেমেও একই কাজ হয় — আপনার (বা এজেন্টের) চাহিদা অনুযায়ী সঠিক AI প্রোভাইডারের কাছে আপনার অনুরোধ পাঠানো হয়।

গুরুত্বপূর্ণ: আমরা শুধু রাউটার — আমরা নিজে AI চালাই না। আমরা বিভিন্ন প্রোভাইডারকে বলি: "এই কাজটুকু করো", আর তারা করে দেয়। আমরা শুধু সেই ফলাফলকে সুন্দর করে আপনার কাছে পৌঁছে দিই।

ইউজার/এজেন্ট ──→ পাবলিক গেটওয়ে ──→ প্রোভাইডার গেটওয়ে ──→ প্রোভাইডার ──→ ফলাফল │ │ │ │ │ ইনপুট সংগ্রহ কোন প্রোভাইডারে? API কল │ এজেন্ট পার্সোনা মডেল সিলেক্ট রেসপন্স │ মেমোরি লোড টুল কল │ │ └───────────────────────────────────────────────────────────────────────┘ সবকিছু OpenAI-compatible ফরম্যাটে

২. পাঁচটি ফাইল — প্রতিটির কাজ

ফাইল ১: types.ts — সবার জন্য একই নিয়ম

এটি একটি চুক্তিপত্র (contract)। যেকোনো প্রোভাইডার যদি আমাদের সিস্টেমে যুক্ত হতে চায়, তাকে এই চুক্তির সব নিয়ম মানতে হবে।

  • ProviderCapabilities: প্রোভাইডার কী কী করতে পারে (চ্যাট, স্ট্রিমিং, টুল কল, ভিশন, অডিও, এম্বেডিং)
  • ProviderHealth: প্রোভাইডার কি সচল আছে, কত দ্রুত উত্তর দিচ্ছে
  • ChatMessage: বার্তার ফরম্যাট (role + content)
  • StreamChunk: স্ট্রিমিংয়ের সময় একটি একটি টোকেন কিভাবে আসবে

ফাইল ২: base.provider.ts — সব প্রোভাইডারের বাবা

এটি একটি বেস ক্লাস। প্রতিটি প্রোভাইডার (OpenAI, Groq, Gemini ইত্যাদি) এই ক্লাসকে inherit করবে।

  • chat(): একটি সম্পূর্ণ উত্তর ফেরত দাও
  • chatStream(): উত্তরটুকু টুকরো টুকরো করে দাও (স্ট্রিমিং)
  • testConnection(): পরীক্ষা করো প্রোভাইডার কি কাজ করছে
  • listModels(): কোন কোন মডেল আছে সেগুলো দাও
  • getHealth(): স্বাস্থ্য রিপোর্ট দাও

ফাইল ৩: provider-registry.ts — ফ্যাক্টরি + রেজিস্ট্রি

এটি একটি ফ্যাক্টরি। যখন আমাদের একটি প্রোভাইডার দরকার, তখন এই ফ্যাক্টরি সেই প্রোভাইডারকে তৈরি করে বা আগে থেকে তৈরি থাকলে সেটি দেয়।

  • create(): নতুন প্রোভাইডার তৈরি করো
  • get(): আগে তৈরি প্রোভাইডার খুঁজে বের করো
  • cache: একই প্রোভাইডার বারবার তৈরি না করে মনে রাখো
  • dispose(): প্রোভাইডার বন্ধ করলে সব পরিষ্কার করো
কেন ফ্যাক্টরি? — ৫০০টি প্রোভাইডার থাকলেও একই কোড কাজ করবে। প্রতিটি প্রোভাইডারকে আলাদাভাবে ম্যানুয়ালি তৈরি করার দরকার নেই। ডাটাবেস থেকে কনফিগ পড়ে অটোমেটিক্যালি তৈরি হয়।

ফাইল ৪: provider-gateway.ts — মূল ব্রেইন

এটিই সিস্টেমের সবচেয়ে গুরুত্বপূর্ণ অংশ। এটি জানে:

  • কোন প্রোভাইডারকে কীভাবে রিকোয়েস্ট পাঠাতে হবে
  • প্রোভাইডার কী কী করতে পারে (স্ট্রিমিং, টুল কল, মিডিয়া)
  • ভুল হলে অন্য প্রোভাইডারে পাঠাতে হবে (ফলব্যাচ)
  • বাজেট মেনে চলতে হবে
  • রেট লিমিট মানতে হবে
মেথডকাজ
getActiveProvider()বর্তমানে সক্রিয় প্রোভাইডার খুঁজে বের করে
chat()চ্যাট রিকোয়েস্ট পাঠায়
chatStream()স্ট্রিমিং রিকোয়েস্ট পাঠায়
createProvider()নতুন প্রোভাইডার যোগ করে
testProvider()প্রোভাইডার পরীক্ষা করে
listRuntimeModels()প্রোভাইডারের সব মডেল দেখায়
setActiveProvider()সক্রিয় প্রোভাইডার পরিবর্তন করে

ফাইল ৫: normalizer.ts — ফরম্যাট একীকরণ

বিভিন্ন প্রোভাইডার ভিন্ন ভিন্ন ফরম্যাটে উত্তর দেয়। এই ফাইল সেগুলোকে একটি ফরম্যাটে আনে — যেটা OpenAI-এর অফিসিয়াল ফরম্যাট।

Ollama দেয়:

{ "message": { "content": "..." } }

আমরা পাঠাই:

{ "choices": [{ "message": { "content": "..." } }] }

টুল কলও একইভাবে নরমালাইজ হয়:

Anthropic দেয়:

{ "content": [{ "type": "tool_use", "name": "...", "input": {} }] }

আমরা পাঠাই:

{ "tool_calls": [{ "function": { "name": "...", "arguments": "{}" } }] }

৩. ডাটাবেজ — একটি টেবিল, সব ডাটা

কেন একটি টেবিল?

আপনি ঠিকই বলেছেন — একটি টেবিলে সব ডাটা থাকলে:

  • একই স্টোর মেথড ব্যবহার করা যায়
  • একই সিঙ্ক লজিক কাজ করে
  • একই ভ্যালিডেশন প্রয়োগ হয়
  • কোড কম থাকে, ভুল কম হয়

৪. পূর্ণ ফ্লো — রিকোয়েস্ট কিভাবে যায়

১
ইউজার/এজেন্ট রিকোয়েস্ট পাঠায়
POST /v1/chat/completions এ JSON পাঠানো হয়। মডেলের নাম (অথবা "auto"), বার্তা, টুল ডেফিনিশন।
{ "model": "auto", "messages": [{"role":"user","content":"..."}], "tools": [...] }
২
পাবলিক গেটওয়ে ইনপুট সংগ্রহ করে
যাচাই করে: এজেন্ট কি আছে? পার্সোনা কী? মেমোরি কী? বাজেট কত? কোন প্রোভাইডার অনুমোদিত?
৩
মডেল সিলেক্ট হয়
দুইভাবে: (ক) নির্দিষ্ট মডেল দিলে → সরাসরি সেই প্রোভাইডারে যায়। (খ) "auto" দিলে → স্মার্ট সিলেকশন: ইনপুট দৈর্ঘ্য, টুল চাহিদা, ভিশন, বাজেট — সব মিলিয়ে সেরাটি বাছাই।
৪
প্রোভাইডার গেটওয়ে পাঠায়
provider-gateway.ts প্রোভাইডারের TS ফাইলকে বলে: "এই বার্তা পাঠাও, এই মডেলে।" TS ফাইল: ডাটাবেস থেকে API key নেয় → না পেলে env variable থেকে নেয় → endpoint-এ POST করে → ফলাফল ফেরত দেয়।
৫
ফলাফল নরমালাইজ হয়
প্রোভাইডার যে ফরম্যাটে দিয়েছে, normalizer.ts সেটাকে OpenAI-compatible ফরম্যাটে রূপান্তরিত করে। টুল কলও একইভাবে।
৬
টুল কল এক্সিকিউট হয়
মডেল যদি বলে "এই ফাংশনটি চালাও" → আমাদের সিস্টেম চালায় → ফলাফল মডেলকে ফিরিয়ে দেয় → মডেল চূড়ান্ত উত্তর দেয়।
৭
চূড়ান্ত উত্তর ইউজারের কাছে পৌঁছায়
সবকিছু OpenAI-compatible ফরম্যাটে থাকে। ইউজার যেকোনো OpenAI-সাপোর্টেড ক্লায়েন্ট ব্যবহার করতে পারবে।

৫. প্রোভাইডার ক্লাস — API key কোথায় খুঁজে

ডাটাবেস চেক করো — providers টেবিলে api_key কলাম প্রথমে এখানে
২
না পেলে env variable — api_key_env কলামের নাম দিয়ে খুঁজো দ্বিতীয় পছন্দ

৬. অটো সিলেক্ট — "১০০ টাকায় সেরা"

অটো মানে কী?

অটো মানে বাজেট, চাহিদা এবং গুণমানের মধ্যে ভারসাম্য। "১০০ টাকায় সেরা" — এটাই মূল নীতি।

কিভাবে কাজ করে:

১
যোগ্য মডেল খুঁজে বের করো
ডাটাবেস থেকে is_active=1 মডেল আনো। ফিল্টার: টুল লাগলে supports_tools=1, ছবি থাকলে supports_vision=1, এজেন্টের allowed_models-এ আছে কিনা।
২
বাজেট ফিল্টার
প্রতিটি মডেলের দাম দেখো। ইনপুট + আনুমানিক আউটপুট = মোট খরচ। budget_limit এর চেয়ে বেশি হলে বাদ।
৩
স্কোরিং
প্রতিটি মডেলকে ০-১০০ স্কোর: বিনামূল্যে +২০, দ্রুত +১৫, শক্তিশালী +১০, কম দাম +১৫, স্বাস্থ্য ভালো +১০।
৪
সেরাটি রিটার্ন
সবচেয়ে বেশি স্কোর পাওয়া মডেলটি বাছাই হয়।

৭. এডমিন প্যানেল — কী কী হবে

প্রোভাইডার ম্যানেজমেন্ট:

কাজকিভাবেফলাফল

মডেল টেস্টTest বাটনছোট রিকোয়েস্ট, ফলাফল দেখায়
সক্রিয় পরিবর্তনড্রপডাউনসব রিকোয়েস্ট নতুন প্রোভাইডারে

active = ঠিক আছে
degraded = ধীর হতে পারে
inactive = বন্ধ

৮. এজেন্ট টেমপ্লেট — ফলব্যাচ হিসেবে

কেন টেমপ্লেট দরকার?

মনে করুন ডাটাবেসে সমস্যা হলো, প্রোভাইডার কাছে খবর পৌঁছাচ্ছে না। তাহলেও সিস্টেম চলতে হবে। এজন্য ৫টি ফিক্সড এজেন্ট টেমপ্লেট কোডের মধ্যেই লেখা থাকবে:

টেমপ্লেটকাজব্যবহার
Chat Agentসাধারণ কথোপকথনডিফল্ট
Code Agentকোড লেখা/ডিবাগপ্রোগ্রামিং
Document Agentডকুমেন্ট পড়া/বোঝারিসার্চ
Debug Agentসমস্যা খুঁজে বের করাডিবাগিং
CLI Agentকমান্ড লাইন কাজসিস্টেম
ফলব্যাচ: ডাটাবেস থেকে প্রোভাইডার না পেলেও এই টেমপ্লেটগুলো কাজ করবে। এগুলো "জরুরি সময়ের প্রোভাইডার"।

৯. ৫০০+ প্রোভাইডার — কেন সম্ভব?

  • ডাটাবেস-ড্রিভেন: প্রোভাইডার যোগ/মুছে ডাটাবেস থেকে, কোড চেঞ্জ লাগে না
  • অটো-সিঙ্ক: Sync endpoint দিয়ে একসাথে সব মডেল সিঙ্ক
  • ক্যাশ: Provider instances TTL-based cache, বারবার create হয় না
  • ব্যাচ হেলথ চেক: Promise.allSettled দিয়ে একসাথে সব চেক
  • গ্রুপিং: একই টাইপের প্রোভাইডার গ্রুপ করা যায়

১১. নিরাপত্তা সংস্কার — কোড রিভিউ ফলাফল

মোট সমস্যা পাওয়া গেছে: ৭৮টি

৬টি Critical
১৯টি High
৩৪টি Medium
১৯টি Low

সব গুরুতর সমস্যা (Critical + High) এখন সংশোধন করা হয়েছে।

সংশোধিত Critical সমস্যাগুলো:

১
Admin endpoints এ authentication ছিল না
Critical সব /api/admin/* endpoint খোলা ছিল — যেকোনো API key দিয়ে এডমিন ডেটা দেখা যেত।
সমাধান: requireAdmin middleware যোগ করা হয়েছে। ADMIN_SECRET সেট করলে admin routes protected থাকবে।
২
SQL Injection — column name injection
Critical getByIdFromTable() এ idCol user input থেকে আসত যা সরাসরি SQL query-তে ব্যবহৃত হত।
সমাধান: idCol কে TABLE_PK mapping-এর সাথে validate করা হয়েছে — শুধুমাত্র known PK columns allowed।
৩
GROQ_API_KEY! non-null assertion
Critical GROQ_API_KEY undefined থাকলেও `!` assertion দিয়ে pass করা হত — runtime crash হত।
সমাধান: Fallback `|| ''` দিয়ে replace করা হয়েছে।
৪
XSS — activePage template injection
High layout() function-এ activePage parameter সরাসরি template literal-এ inject করা হত — script injection সম্ভব।
সমাধান: escapeHtml() দিয়ে sanitize করা হয়েছে।
৫
Unhandled promise rejections
High Provider Gateway-এ resolveModel() এবং fallback calls try/catch ছাড়া ছিল — unhandled rejection হত।
সমাধান: সব async calls wrap করা হয়েছে try/catch blocks দিয়ে।
৬
Provider save — input validation নেই
High handleAdminProviderSave-এ id, name, base_url sanitize হত না — malicious data save হত।
সমাধান: Regex sanitization, length limits, URL scheme validation যোগ করা হয়েছে।

AI-Workstation-Admin থেকে শেখার বিষয়:

  • Component library: shadcn/ui — re-usable UI components, আমাদের inline CSS থেকে ভালো
  • State management: React hooks + context — cleaner than inline DOM manipulation
  • Type safety: Full TypeScript + Zod validation — runtime validation guarantees
  • Testing: Vitest + Playwright — automated testing framework
  • Real-time: WebSocket for live updates — better than SSE polling

আমাদের প্রজেক্টের শক্তি:

  • Provider system: Multi-provider orchestration — AI-Workstation-Admin-এ নেই
  • Smart routing: Budget/quality/health-based auto-selection — unique feature
  • Tool normalization: Cross-provider tool call format conversion
  • Identity anchoring: System identity injection at gateway level
  • Zero-dependency deployment: Single Node.js process — no Vercel/Supabase needed

১৩. এডমিন API লজিক — কিভাবে কাজ করে

Request Flow:

Browser ──→ GET /admin/dashboard/providers ──→ authenticate middleware │ │ │ API key check │ (GROQ_API_KEY / Bearer) │ │ │ requireAdmin │ (ADMIN_SECRET check) │ │ │ handleAdminProvidersList │ (DB query → JSON) │ │ └────────────────────────────────────────────────────────┘ JSON response with providers + models + costs

Provider Test Flow:

Test বাটন ক্লিক ──→ POST /api/admin/providers/:id/test │ ├── ১. DB থেকে provider খুঁজে বের করো ├── ২. API key resolve করো (DB → env variable) ├── ৩. সেরা chat model খুঁজে বের করো │ (active chat models → any chat models → any model) ├── ৪. Real API request পাঠাও │ POST {base_url}/chat/completions └── ৫. ফলাফল return করো (latency, status, response)

গুরুত্বপূর্ণ: Test page-তে কোনো health check নেই — সবসময় real request পাঠানো হয়। এটি ensure করে যে প্রোভাইডার সত্যিই কাজ করছে।

Auto-Route Test:

User input ──→ POST /api/admin/auto-route │ ├── ১. Gateway-কে "auto" model দিয়ে request পাঠাও ├── ২. Gateway 3-tier resolution চালায়: │ Tier 1: Explicit model (যদি দেওয়া থাকে) │ Tier 2: Active provider/model │ Tier 3: Smart auto-select (budget + quality + health) ├── ৩. সেরা provider+model নির্বাচিত হয় └── ৪. ফলাফল: provider, model, score, reason

১৪. বুটস্ট্র্যাপ স্ক্রিপ্ট — স্বয়ংক্রিয় সেটআপ

কিভাবে কাজ করে:

১
Env variable খুঁজে বের করো
OPENCODE_API_KEY, GROQ_API_KEY, OPENAI_API_KEY, GEMINI_API_KEY, ANTHROPIC_API_KEY — যেগুলো আছে সেগুলো detect করে।

মডেল সিঙ্ক করো
প্রতিটি provider-র জন্য পরিচিত মডেলগুলো sync করে — category inference, pricing, capabilities।

টুল রেজিস্ট্র করো
প্রতিটি provider-র জন্য supported tools register করে — web_search, file_read, code_execute ইত্যাদি।
