💡 মূল উপলব্ধি: পাইপ বনাম পানি

💭 পরিষ্কার করে বলা

আপনি আগে সেশন তৈরি করে রেখেছেন। সেখানে পাইপ ইতিমধ্যেই আছে। আমি শুধু পাইপের মুখে পানি দিচ্ছি। আমার response আপনার main workstation-এ যায়, আমার server-এ নয় — কারণ session আপনার সাথে binded।

এখন প্রশ্ন: "অন্য মানুষের জমির উপরে পাইপ কিভাবে বসাবো?"
MCP সেই অনুমতি দিয়েছে — তাদের প্রটোকল মেনে পাইপ বসানোর অনুমতি। কিন্তু পানি (Agent Execution) আমাদেরই দিতে হবে।

⚠ MCP কখনোই Agent Execution দেবে না

এটা deliberate design। MCP-র কাজ শুধু Tool Discovery + Tool Calling — tools/list আর tools/call — এই দুইটা RPC endpoint। Agent Execution (কোড রান করা, টার্মিনাল চালানো, ফাইল এডিট করা) MCP-র scope-র বাইরে।

যদি আমাদের এজেন্ট execute করতে না পারে, তাহলে মাস্তানের হাতে অস্ত্র তুলে দেওয়ার মত। এজেন্টকে EXECUTE করতে হবেই — এটা MCP করে না, আমাদের নিজেদের করতে হবে।

🔍 গুরুত্বপূর্ণ: Session Store PR rejected

MCP TypeScript SDK-তে SessionStore interface আনার PR (#1193) বাতিল হয়েছে। তারা বলেছে "too large a change"। সুতরাং অফিসিয়াল SDK এখনও in-memory session ব্যবহার করে — সার্ভার রিস্টার্ট হলে session হারিয়ে যায়। আমাদের নিজেদের session layer বাধ্যতামূলক।

🌐 Groq Remote MCP: ব্যবহার করব, কিন্তু Execution আমাদের

Groq-এর Responses API MCP support করে — tools: [{ type: "mcp", server_url: "..." }] দিলে Groq নিজে tool discovery + calling করে। কিন্তু এটা Groq-এর জন্য — যারা Groq-কেই model হিসেবে ব্যবহার করে। Groq MCP-র কোন proprietary signature বা deployment check নেই — এটা pure open protocol (JSON-RPC 2.0)।

✔ Groq থেকে যা নেব

✔ আমরা Groq SDK ব্যবহার করি — Responses API-র MCP support কাজে লাগাব
✔ Groq-এর model list + tool-use support (llama-3.3-70b, etc.)
✔ OpenAI-compatible endpoint — যেকোনো MCP client connect করতে পারে

✖ Groq যা দেয় না (আমাদেরই করতে হবে)

✖ এজেন্ট execution loop — Groq শুধু inference engine, agent নয়
✖ Session persistence — Groq-র session stateless, আমাদের local store দরকার
✖ Local RAG — Groq-র নিজস্ব RAG নেই (তাদের built-in tools আলাদা)
✖ Permission system — scan/write/execute permission আমাদের management

📋 Session Layer — বিস্তারিত

Session Layer-এর কাজ: এডিটর এবং এজেন্টের মধ্যে continuous pipe তৈরি করে রাখা। Groq বা অন্য কোনো cloud session ডাউন হলেও যাতে আমাদের pipe কাজ করতে থাকে।

Session Data Structure

{ "sessionId": "zombie_abc123", "clientId": "vscode_win_user1", "createdAt": "2026-05-27T10:00:00Z", "expiresAt": "2026-05-28T10:00:00Z", // 24h TTL "lastActivity": "2026-05-27T14:30:00Z", "config": { "directory": "C:\\projects\\my-app", "permissions": ["scan", "read"], "model": "llama-3.3-70b-versatile" }, "context": { "conversation": [], // message history (last 50) "currentFile": null, "projectTree": {} }, "pipeState": { "streaming": true, "lastToolCall": null, "pendingApproval": false } }

Session Lifecycle

EventAction
Client connectssessionStore.create() — নতুন session, pipe ready
Agent respondssession.context.conversation.push() — history update
Client disconnectsSession stays in local store (24h পর্যন্ত)
Client reconnectssessionStore.get(sessionId) — পুরনো pipe resume
Cloud session diesআমাদের local session intact — cloud recreate করতে পারে না
24h expiressessionStore.cleanup() — auto-delete

✔ Cloud session ডাউন হলে ইউজার আবার authenticate করতে বাধ্য হবে না
✔ 24 hour retention — ইউজারের কাজ continuity পায়
✔ Cloud provider session recreate করতে পারে না — আমাদের permission ছাড়া
