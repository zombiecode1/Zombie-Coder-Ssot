Proxi Agent সিস্টেম: টুল, এজেন্ট, সেশন, মেমরি, রাউটিং

এই পেজটি প্রজেক্টের মূল ফাইল ও সার্ভিসের কাজ, ত্রুটি উৎস, এবং বাস্তব ব্যবহারের ব্যাখ্যা বাংলায় দেয়া হয়েছে।

১. প্রধান সমস্যা

রিপোর্ট করা ত্রুটি:

Provider source 'opencode' is unavailable

এবং VS Code JSON schema warning: Unable to load schema from 'https://opencode.ai/config.json': Location ... is untrusted.

এটার আসল মানে হচ্ছে:

  • opencode.json ফাইলটি $schema হিসেবে https://opencode.ai/config.json ব্যবহার করছে।
  • VS Code সেটাকে “untrusted” হিসেবে দেখছে, তাই স্কিমা লোড হচ্ছে না।
  • রানটাইমে প্রকৃত provider হিসেবে
  • proxi-bridge ব্যবহার হচ্ছে, কিন্তু কোডে opencode-zen ডিফল্ট ব্যাকএন্ড হিসাবে সেট আছে।

২. opencode.json ও .opencode/opencode.json কী করছে

এই ফাইল দুটি একদম একই কনফিগারেশন শেয়ার করে। মূল বিষয়:

  • provider.proxi-bridge ব্যবহার করে, যে URL দিচ্ছে http://localhost:9999/v1.
  • এখানে মডেল তালিকা দেয়া আছে, যেমন llama-3.3-70b-versatile, qwen/qwen3-32b, openai/gpt-oss-120b ইত্যাদি।
  • mcp.mcp-agent-server remote MCP সার্ভার হিসাবে http://localhost:3000/mcp ব্যবহার করছে।
  • এজেন্ট সংজ্ঞা দুইটি আছে: zombie-coder ও zombie-explorer।
মন্তব্য: এখানে provider হিসাবে opencode নয়, proxi-bridge ব্যবহার হচ্ছে। তাই runtime তে যদি opencode রুট খোঁজার চেষ্টা হয়, তা ভুল বা mismatch।

৩. src/wrapper/config.ts আর ব্যাকএন্ড রেজোলিউশন

এই ফাইল নির্ধারণ করে কোন ব্যাকএন্ড থেকে মডেল চালানো হবে:

  • local-bridge: ডিফল্ট এবং সবচেয়ে আগে চেক করা হয়, URL http://localhost:9999/v1.
  • opencode-zen: দ্বিতীয় ব্যাকএন্ড, URL https://opencode.ai/zen/v1 এবং OPENCODE_API_KEY
  • defaultBackend ডিফল্টভাবে opencode-zen, কিন্তু resolveModel() প্রথমে zen মডেল চেক করে, তারপর bridgefallback করে।

এখানে সমস্যা: যদি OPENCODE_API_KEYopencode.ai ব্লক করা থাকে, তাহলে প্রথমত opencode-zen ব্যাকএন্ডকে ব্যবহার করার চেষ্টা হতে পারে এবং সেই সময় “provider unavailable” বা অ্যাক্সেস ত্রুটি উঠে আসতে পারে।

if (zen && zen.models.includes(model)) {
  return { backend: zen, modelName: model }
}

if (bridge && bridge.models.includes(model)) {
  return { backend: bridge, modelName: model }
}

এই লজিকে যদি মডেলটি opencode-zen-এর তালিকায় থাকে, কিন্তু OPENCODE_API_KEY

সুস্পষ্ট ভুল: ডিফল্ট সেকশন defaultBackend: 'opencode-zen' কিন্তু বাস্তবে আপনার opencode.json ব্যাকএন্ড হল proxi-bridge। এই mismatch runtime-এ বিভ্রান্তি তৈরি করতে পারে।

৪. src/controllers/mcpController.ts: MCP সেশন, টুল, রিসোর্স, রাউট

এই ফাইলটি মূলত MCP-সাপোর্টেড JSON-RPC সার্ভার হিসেবে কাজ করে। প্রধান কাজ:

  • সেশন তৈরি ও টাচ করা (initialize, touchSession)
  • সেশনের টিএলএল: ৩০ মিনিট জীবিত রাখা
  • `buildTools()` ফাংশন যে টুলগুলো প্রকাশ করে:
workspace_index, workspace_search, conversation_create,
conversation_history, conversation_list, ssot_read, project_status, agent_routes

এই মানে হলো: MCP ক্লায়েন্ট একাধিক পরিষেবা একসাথে ব্যবহার করতে পারে — ওয়ার্কস্পেস ইনডেক্স, সার্চ, কনভারসেশন লিস্ট, SSOT পড়া, এজেন্ট স্ট্যাটাস, এজেন্ট রাউট।

রাউটিং সম্পর্কিত অংশ:

  • agent_routes নামক টুল আছে, যা `getMawlanaRouter()?.getAllRoutes()` থেকে রুট তালিকা ফেরত দেয়।
  • এই রুটগুলো active agent decision বা subagent route নির্বাচন দেখাতে পারে।
  • এখানে `resources` ও `status` ফিল্ড আছে, যেমন zombiecoder://status, zombiecoder://ssot ইত্যাদি।

অর্থাৎ, MCP সাপোর্টের প্রধান ফোকাস:

  1. সেশন লাইফসাইকেল ও ট্র্যাকিং
  2. সেশন ভিত্তিক লগ স্তর ও ক্লায়েন্ট তথ্য
  3. agent-র রাস্তাগুলি প্রকাশ করা
  4. SSOT ও স্ট্যাটাস রিডিং

৫. src/index.ts ও সার্ভার স্টার্টআপ

এই ফাইলটি সার্ভিস চালু করে এবং runtime manifest লিখে:

  • এক্সপ্রেস অ্যাপ বানায়, CORS ও JSON পার্সার ব্যবহার করে।
  • `/v1`, `/api`, `/dashboard`, `/admin`-এ authentication যোগ করে।
  • `initializeAgentSystem(DEFAULT_WORKSPACE)` কল করে।
  • প্রিন্ট করে যে agent ও model endpoints কী।

এখানে `runtime.json` তৈরি করে, যেখানে থাকে:

  • workspaceRoot
  • mcpUrl ও sseUrl
  • editor config path

এই ফাইল দেখায় যে প্রকৃত উদ্দেশ্য হলো একটি OpenAI-Compatible ব্রিজ এবং একইসঙ্গে ZombieCoder এজেন্ট সিস্টেম একসাথে চালানো।

৬. মেমরি / সেশন / রাউটিং বাস্তবে কী কাজ করে

এই প্রজেক্টের মূল বৈশিষ্ট্যগুলো:

  • মেমরি: এখানে সরাসরি মেমোরি স্টোর নেই, তবে MCP সেশন সংরক্ষণ করে mcpSessions ম্যাপে।
  • সেশন: ক্লায়েন্ট initialize করলে নতুন sessionId তৈরি হয়, response-এ Mcp-Session-Id পাঠায়।
  • রাউটিং: agent routes endpoint এজেন্টের সিদ্ধান্ত ও সাবএজেন্ট রাউট দেখায়।
  • টুলস: JSON-RPC মেকানিজমে গ্রাহককে tools এবং resources উপলব্ধ করে।
সারসংক্ষেপ: এই সিস্টেমের কাজ হল মেমরি নয়, বরং সেশন পরিচালনা, agent routes প্রকাশ করা এবং MCC প্রোটোকল ভিত্তিক ক্লায়েন্ট/সার্ভার ইন্টিগ্রেশন নিশ্চিত করা।

৭. কি ঠিক আছে, কি বদলাতে হবে

ঠিক আছে:

  • opencode.json ও .opencode/opencode.json একই কনফিগারেশন শেয়ার করছে।
  • src/controllers/mcpController.ts সেশন, টুলস, resources, এবং স্ট্যাটাস সার্ভিস কভার করছে।
  • src/index.ts ওপেনএআই-কম্প্যাটিবল ব্রিজ সার্ভার এবং agent সিস্টেম একসঙ্গে চালানোর পরিকল্পনা দেখায়।

বদলাতে হবে:

  • src/wrapper/config.ts থেকে যদি বাস্তব রিয়েল-টাইম ডিফল্ট হয় proxi-bridge, তাহলে defaultBackend আপডেট করা উচিত।
  • `opencode.json` এর $schema কেউ সরাসরি লোড করতে বাধা পেলে, IDE warnings আসবে; এটা সারফেস এনালাইসিস এ ছাড়া runtime-এ বাধা দেয় না।
  • যদি opencode-zen ব্যাকএন্ড ব্যবহার করা হয়, তবে OPENCODE_API_KEY আবশ্যক। না থাকলে config রুটিন নিশ্চতভাবে fallback করতে হবে।
  • `provider source 'opencode' unavailable` মেসেজ হলে প্রথমে চেক করতে হবে যে ব্যাকএন্ডের URL/কী ঠিক আছে নাকি provider registration ভুল।

৮. নিরাপদ সুপারিশ

  1. IDE স্কিমা সমস্যা সমাধান
    VS Code-এ $schema লোড না হলে `opencode.ai` ব্লক হতে পারে। এটি স্কিমা লোড সমস্যা, প্রকৃত ক্লাউড সার্ভিস সমস্যা না।
  2. ব্যাকএন্ড মিলানো
    `opencode.json`-এ provider হিসেবে proxi-bridge ঠিক আছে, তাই runtime config-এও ডিফল্ট ব্যাকএন্ড একই রাখা উচিৎ।
  3. মডেল রাউটিং বৈধতা
    `resolveModel()`-এ প্রথমে ব্রিজ, তারপর opencode-zen ফর্ক ব্যবহার করলে runtime রাউটিং বেশি স্থিতিশীল হবে।
  4. MCP সেশন মনিটর
    `mcpController.ts` তে লস্ট সেশন বা SSE ট্র্যাকিং যাচাই করতে হবে, কারণ session-ভিত্তিক agent operation এখানে নির্ভর করে।
